# WORKFLOW-DISCOVERY.md

**Every finding below is from direct source inspection, not assumed from memory of having built these systems across Sprints 1-10.**

## What Already Exists (Confirmed)
window.autoGenerateTask() (Sprint 1), 6 pre-existing trigger types (vaccination_scheduled, expected_birth_approaching, medication_followup, weight_alert, health_risk_alert, production_alert), all with deterministic dedup on eventType+':'+sourceId. evaluateWeightAlert()/evaluateProductionAlert() (Sprint 2/4) already self-resolve correctly -- confirmed 4 real status:'resolved' writes when the underlying condition clears. evaluateHealthRisk()/evaluateOperationalPriority() (Sprint 3/5) are pure, read-only, no persisted state, so no staleness is even possible for them.

## The Real Gap This Discovery Found: One-Time Reminders Never Get Closed
Three domains create a reminder task for a future event, but nothing anywhere marks that task done when the event it was reminding about actually happens:

| Domain | Reminder created when... | What SHOULD close it | What actually closes it (confirmed) |
|---|---|---|---|
| Birth | breeding.status becomes 'pregnant' (expected_birth_approaching) | The birth is actually recorded (status becomes 'born', createOffspringAnimal() runs) | Nothing. createOffspringAnimal() (shared.js) creates the animal + weight record only -- confirmed by reading its full body, zero task interaction |
| Vaccination | A vaccination is scheduled 'pending' (vaccination_scheduled) | The vaccination is marked 'done' | Nothing. pages/vaccine.js's status-update path has no autoGenerateTask/task-resolution call at all |
| Medication | health.withdrawal_end is set (medication_followup) | The withdrawal period passes, or the health record is marked complete | Nothing. Confirmed by reading the full submitHealth() write path |

Consequence, confirmed by direct reasoning about the dedup mechanism: these reminder tasks sit as pending indefinitely -- a farm worker sees "birth expected" on their task list for an animal that already gave birth days ago, unless someone manually opens tasks.html and marks it done by hand. This is the concrete "multiple manual steps that could become one workflow" this sprint's mission names directly.

## Operations With Zero Automation (Confirmed, Not Assumed)
Sale (animals.html, animal-detail.html -- 3 write sites, status:'sold') and Death (animal-detail.html, status:'dead', died_at) have zero automation hooks anywhere -- confirmed via exhaustive search for autoGenerateTask near either write path. Neither closes any outstanding reminder task tied to that animal (a sold or dead animal can still show up with a pending vaccination reminder, for instance) nor generates any follow-up.

## Operations That Are Already Fully Automated (No Gap)
Weight recording -> evaluateWeightAlert() (self-resolving). Production recording -> evaluateProductionAlert() (self-resolving). Health recording -> feeds evaluateHealthRisk() live (nothing to resolve, it's not persisted). Notification acknowledgement (markRead, Sprint 9) and task completion (pages/tasks.js) are themselves terminal actions -- neither currently triggers any follow-up recommendation, which Phase 4 of this sprint addresses.

## Flow Map (Representative -- Vaccination, Chosen Because It Shows Both the Working Chain and the Gap)

```
User Action (mark vaccination 'done' on vaccine.html)
        |
        v
Generated Record (vaccinations/{id}.status = 'done')          <- SSOT write, unchanged
        |
        v
Notification -- NONE generated by this specific transition today (confirmed: no
                notifications-service.js trigger fires on a 'done' transition,
                only on 'overdue'/'pending-soon' states)
        |
        v
Task -- the ORIGINAL 'vaccination_scheduled' reminder task from when this
        vaccination was first scheduled REMAINS PENDING -- this is the gap
        |
        v
Reports/Analytics -- Sprint 9's notification stats and Sprint 10's
        vaccinationsCompleted count DO correctly reflect the completion
        (both read status/done_date directly, unaffected by the task gap)
        |
        v
Dashboard -- Executive KPI Strip's compliance % updates correctly (same reason)
```

The gap is narrow and specific: the task list, not the data itself, is what goes stale. Reports/Analytics/Dashboard already read the underlying records directly and are correct; only the human-facing reminder fails to clear itself.
