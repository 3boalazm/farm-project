# D-03: AI Birth Parity

**Status:** APPROVED, IMPLEMENTED (Wave B, Commit 2)

**Decision:** AI Assistant's `add_birth` intent becomes a thin caller of the canonical Birth workflow — creating a `breeding` record and reusing `createOffspringAnimal()` for each offspring.

**Context:** `add_birth` created animal records via its own independent `fbPost('animals',...)` call, with no corresponding `breeding` record and no weight-history entry — both confirmed technical debt, not stated product choices. Two additional gaps (`father_tag`, `tag` support) are unresolved product decisions, lower-stakes, explicitly not blocking this convergence.

**Consequences:** weight history and breeding-linkage become real for AI-recorded births; one creation pattern instead of two for the animal-record portion.

**Historical Strategy:** None required — forward-only, identical in spirit to D-02. Existing AI-created records remain as they are.

**Deferred:** `father_tag`/`tag` support — the AI's own payload schema has no source field for either; not invented.

**Implementation:** See `docs/certification/BIRTH.md`.
