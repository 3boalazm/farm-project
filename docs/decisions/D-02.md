# D-02: Birth SSOT via markBorn

**Status:** APPROVED, IMPLEMENTED (Wave B, Commit 1)

**Decision:** `markBorn` (`submitBreeding()`, `status='born'`) creates real `animals` documents, matching `_ubSubmit`'s pattern, going forward only.

**Context:** `markBorn` previously stored only aggregate counts (`offspring_count`, `male_offspring`, `female_offspring`) with zero corresponding animal documents — structurally identical to D-01's original weight-storage problem, and the direct cause of `pedigree.js` being unable to trace these births at all (genealogy requires real `mother_tag`/`father_tag` on real animal documents).

**Evidence considered:** `pages/reports.js`, `assistant.html`'s AI summary, `import.html`, and `animal-detail.html` all directly consume `offspring_count` as a trusted number — explicitly *not* touched by this decision's implementation, to keep the change narrowly scoped.

**Consequences:** genealogy, search, weight-history, and health-record capability become possible for `markBorn`-created animals; `offspring_count` remains present but is not the sole source of truth going forward.

**Historical Strategy:** Leave historical `markBorn` records untouched. Individual offspring identity, exact per-animal birth weight, and gender-specific tagging were never captured at the time — any synthesized backfill would manufacture false-precision records rather than real history.

**Implementation:** extracted `createOffspringAnimal()` shared helper (also adopted by `_ubSubmit()`, behavior-preserving); duplicate-prevention only fires on genuine transition into `born` status. See `docs/certification/BIRTH.md`.
