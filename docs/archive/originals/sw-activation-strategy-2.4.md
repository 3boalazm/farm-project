This is a recommendation for a future implementation task, not something built here.

## PHASE 2 — Rollout Strategy

### Option A — Enable immediately for all users
- **User impact:** Every user, web and Android, is affected on their very next page load after deployment — no gradual exposure.
- **Failure scenarios:** Because `skipWaiting()`/`clients.claim()` are already present (unmodified, per this task's constraints), the very first activation takes effect immediately, for 100% of users simultaneously, with no built-in staging. Any unforeseen interaction (e.g., a caching edge case not caught in testing) affects the entire user base at once.
- **Rollback complexity:** High relative to the other options specifically *because* the blast radius is total — every user needs the same corrective new SW version pushed and re-activated (per the established rollback model: a new version, not a plain file revert) before the issue is resolved for anyone.
- **Monitoring needs:** Would need to be watching universally from the moment of deployment — no "canary" window to catch a problem before it's already affecting everyone.

### Option B — Controlled rollout / staged activation
- **User impact:** A subset of users/sessions activate the service worker first; the rest continue exactly as today (unregistered) until the staged group is confirmed healthy.
- **Failure scenarios:** Contained to the initial subset — a problem is discovered while affecting only a fraction of users, not everyone.
- **Rollback complexity:** Lower — the affected group is smaller, and (importantly) the majority of users who were never part of the staged rollout never had a service worker to roll back in the first place; only the staged subset needs the corrective new-version treatment.
- **Monitoring needs:** Still requires monitoring, but with a natural checkpoint — no need to watch the entire user base with the same urgency from second one, since the exposure is already limited by design. This project doesn't currently have server-side user-segmentation infrastructure (per the earlier Refactoring Blueprint's Release Strategy discussion), so "staged" here would practically mean something simple and client-side — e.g., a flag checked before calling `register()`, decided once per session/device rather than a true server-controlled percentage rollout. That's an implementation detail for a future task, not decided here.

### Option C — Delay activation until more testing
- **User impact:** None — no change from today's already-long-standing status quo (fully-built, unregistered).
- **Failure scenarios:** None introduced; the risk is entirely opportunity cost — the offline capability this project's own APK release-notes template already advertises continues to not exist for any real user, with no defined timeline.
- **Rollback complexity:** N/A — nothing is live to roll back.
- **Monitoring needs:** None beyond whatever manual QA precedes the eventual decision to move to A or B.

**No single option is being recommended as final here** — Phase 2 explicitly asks for a comparison, and the right choice depends on factors outside this technical investigation (how much manual testing has actually happened, whether the team wants a canary mechanism built first). What this investigation can say directly: **given Option A's specific combination of `skipWaiting`+`clients.claim` (immediate, total takeover with no native staging) and this project's current lack of a rollback mechanism simpler than "ship a new version," Option B or C carry meaningfully lower first-activation risk than Option A**, independent of any other consideration.

## PHASE 3 — Lifecycle Safety Review

- **`skipWaiting()` (present in both the `install` handler and the `message` handler for the `SKIP_WAITING` command):** Forces a newly-installed worker to activate immediately rather than waiting for all existing tabs running a prior worker version to close. For the very **first-ever** registration specifically, there is no prior worker to wait for at all — so this specific call has no observable effect on that first activation (there's nothing to "skip waiting" past yet). Its real effect is on **every subsequent update** after this first activation: each future deploy's new worker takes over immediately rather than waiting for open tabs to close naturally.
- **`clients.claim()` (in `activate`):** Causes the newly-active worker to immediately begin controlling *already-open* pages, not just future page loads. Combined with `skipWaiting()`, this is the "take over everywhere, right now" pattern rather than the more conservative default (a new worker only controls pages loaded *after* it activates, and only takes over previously-open tabs once they naturally reload).
- **First activation behavior:** The page that triggers registration itself was already loaded *before* any service worker existed for it, so that specific page load is not retroactively placed under SW control — only its *subsequent* fetches (AJAX calls, background data refreshes) and any *new* navigation from that point forward are intercepted. This is standard, expected behavior, not a special risk of this specific `sw.js`.
- **Cache takeover behavior:** Once activated, all matching fetches immediately follow the strategies already documented in this Sprint's earlier audits (Cache-First for CDN, Stale-While-Revalidate for HTML/JS/CSS, Network-First-with-fallback for Firebase GET) — there's no gradual "warm-up" period; the moment `activate` completes, every intercepted request is served through these rules.
- **Update behavior (for any future `sw.js` change after this first activation):** A new deploy with bumped cache-version strings triggers `install` for the new worker → (because of `skipWaiting`) it activates immediately, without waiting for existing tabs → (because of `clients.claim`) it immediately starts controlling those already-open tabs too. **A real, if narrow, risk here:** a tab that has old JS/CSS already loaded in memory (from before the update) could have some of its *subsequent* fetches served by the *new* worker's caching rules while its already-executing JavaScript still reflects the *old* version — a brief window of version-mismatch is possible for a page left open across a deploy, though this is a known, general tradeoff of the `skipWaiting`+`clients.claim` pattern industry-wide, not a defect specific to this implementation.
- **Stuck worker scenarios:** Precisely because of `skipWaiting`+`clients.claim`, this project's design is **less** prone to the classic "user stuck on an old cached version indefinitely" failure mode than the conservative default lifecycle would be — updates take effect immediately rather than waiting indefinitely for every tab to close. The tradeoff, as above, is a brief version-mismatch window instead of indefinite staleness — a deliberate, reasonable choice already baked into the existing code, not something this task is changing.
- **Rollback strategy:** Unchanged from every prior task's finding in this chain: rolling back an already-registered, already-active service worker is **not** a simple file revert. It requires shipping a **new** version (a further cache-version bump) that either reverts the problematic logic or explicitly clears caches via the already-existing `CLEAR_DATA_CACHE` message handler — that handler exists specifically to support this scenario, but nothing in the current codebase calls it from the page side yet (confirmed in Task 2.3.2), meaning "rollback" today would rely on the version-bump-and-reactivate mechanism alone, not the message channel, unless page-side code to trigger that message is also built first.

## PHASE 4 — Production Readiness Checklist

| Test | What to verify | Why it matters here specifically |
|---|---|---|
| **Chrome desktop test** | Registration succeeds; `chrome://serviceworker-internals` (or DevTools → Application → Service Workers) shows the worker as activated; Cache Storage shows the three expected named caches populated with the current `APP_SHELL` entries | Chrome is the most common, best-instrumented environment for first-pass verification of the install/activate lifecycle itself |
| **Mobile browser test (Chrome for Android, at minimum)** | Same registration/activation checks as desktop, plus: confirm the app still functions correctly when the browser tab is backgrounded and resumed (a common mobile pattern that can interact with `clients.claim`) | Mobile browsers handle backgrounding/lifecycle events differently from desktop; this project's primary real-world usage is plausibly mobile-heavy given the farm-management context |
| **Android WebView test (inside the actual Capacitor-built APK)** | Registration succeeds inside the WebView specifically (not just a mobile browser) — confirm via remote debugging (`chrome://inspect` against the device) that the SW registers, activates, and that `farm-apk/www`'s now-synced `sw.js` (identical to root, per Task 2.3.5) behaves identically inside the app shell | This is the one environment most likely to have overlooked WebView-specific quirks (e.g., version-dependent Service Worker support on older Android WebView builds) — direct, on-device verification is the only reliable check here, not an assumption from web testing alone |
| **Offline navigation test** | With the device/browser network disabled: (a) previously-visited pages load from cache correctly; (b) a genuinely never-visited, non-`APP_SHELL` page correctly falls back to `offline.html` rather than a raw browser error (this is the specific fallback wired in Task 2.3.4); (c) `offline.html` itself renders correctly with no broken styling (it depends only on the already-cached `styles.css`, per its design) | Directly validates the specific readiness work completed in Tasks 2.3.3–2.3.5 actually behaves as designed under real offline conditions, not just in code review |
| **Cache update test** | Deploy a deliberately trivial change (e.g., to a low-risk file), confirm the next visit picks up the change (Stale-While-Revalidate's "one load behind, then caught up" behavior) rather than staying permanently stale; confirm old-version caches are actually deleted after the new version activates (per the `activate` handler's cleanup logic) | Directly validates the version-bump-and-cleanup mechanism this whole activation strategy depends on for safe future updates, before real users depend on it working correctly |
| **Rollback test** | Deliberately ship a "bad" test version, then exercise the actual rollback mechanism (a further version bump reverting the change) end-to-end, confirming affected clients recover fully | This is the one test category explicitly called for by every prior audit's repeated finding that rollback here is **not** a simple git revert — it should be proven to work as a real, rehearsed procedure before any real incident ever requires it for the first time under pressure |

**Every row above is a recommended future test, not a claim that any of them have been performed as part of this investigation.**

## Output Summary

**1. Recommended registration architecture:** A small, dedicated registration function, invoked from `shared.js`'s existing boot sequence rather than written inline into `shared.js` — universal coverage without deepening that file's already-flagged complexity, and consistent with this Sprint's own Refactoring Blueprint direction.

**2. Recommended rollout strategy:** No single option is mandated by this investigation alone, but **Option A (immediate, universal activation) carries the highest first-activation risk of the three**, specifically because of the already-present `skipWaiting`/`clients.claim` combination and this project's current lack of any staging mechanism — Option B or C are lower-risk starting points if minimizing first-activation blast radius is a priority.

**3. Lifecycle risks:** The existing `skipWaiting`/`clients.claim` pattern (unchanged by this task) trades "indefinite stuck-on-old-version risk" (avoided) for "brief version-mismatch window on future updates for tabs left open across a deploy" (accepted) — a reasonable, already-made design choice, not a new risk introduced by activation itself. The first activation specifically has no unusual risk beyond what's inherent to any first-ever SW registration.

**4. Rollback plan:** Requires a new, superseding `sw.js` version (bumped cache names) rather than a plain file revert — the existing `CLEAR_DATA_CACHE` message handler is designed to support this but has no page-side caller yet, which is itself worth building before relying on it during a real incident.

**5. Production checklist:** Six test categories defined above (Chrome desktop, mobile browser, Android WebView specifically, offline navigation, cache update, and — critically — a rehearsed rollback test), none yet performed.

**6. Approval required before any registration code is written, any activation occurs, or any deployment happens.**
