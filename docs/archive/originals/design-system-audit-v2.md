### اكتشاف معماري جديد — نظام ألوان مستقل ومتوازٍ (Parallel Token System)

عند فحص أعلى القيم "الجديدة تمامًا" تكرارًا (`#f97316`×11, `#3b82f6`×10, `#4caf50`×8, `#22c55e`×8, `#eab308`×7)، تبيّن أنها **مش عشوائية** — كلها موجودة حصريًا في ملفين اثنين فقط: `bayan.html` و`bayan-offline.html`.

**الدليل الكامل:**
```
bayan.html  — سطر 19:  :root{ --green:#22c55e; --blue:#3b82f6; --orange:#f97316; --yellow:#eab308; --purple:#a855f7; }
bayan-offline.html — سطر 12: نفس التعريف بالحرف
```
كلا الملفين عندهم `<style>` block خاص بيهم جوه نفس الملف، **ومبيربطوش `styles.css` الرئيسي إطلاقًا** (تأكدت بالفحص: صفر نتيجة بحث عن `styles.css` جوه الملفين). كمان فيهم `BREED_COLORS` map مستقل (`{'شامي':'#f97316','برقي':'#22c55e',...}`) بيُستخدم في تلوين الرسوم البيانية والفلاتر الخاصة بالسلالات — الاستخدام متسق داخليًا (نفس اللون لنفس السلالة عبر الملف كله).

**ليه ده مهم:** هذا مش "درifت" أو تكرار عرضي — ده **قرار معماري متعمّد** (ولو غير موثّق): صفحات `bayan*` مصمّمة كمستندات مستقلة/قابلة للطباعة (statement pages رسمية)، فبُنيت self-contained من غير اعتماد على ملف CSS خارجي. النتيجة: **يوجد فعليًا نظاما ألوان منفصلان في المشروع** — نظام `styles.css` (التطبيق التفاعلي) ونظام `bayan*.html` الداخلي (المستندات الرسمية) — وهما لا يتشاركان أي توكين مشترك رغم تشابه بعض القيم بالصدفة (`--green` في الاتنين قيمته مختلفة فعليًا: `#059669` في التطبيق مقابل `#22c55e` في بيان).

### اكتشاف إضافي — ملف فارغ تمامًا
`activity.html` = **0 سطر** (ملف فاضي بالكامل). موجود في المشروع كمسار لكن بلا أي محتوى.
`index.html` = 16 سطر فقط، وهو صفحة تحويل (redirect) بحتة بتقرأ `localStorage` وتوجّه لـ `dashboard.html` أو `login.html` — مفيهاش UI حقيقي فتغيب `dir="rtl"` منه بشكل غير مؤثر فعليًا.

### Foundations → Pages Map (ملخص الطبقات)

```
Foundation  → 39 قيمة hex فريدة كمتغيرات (نظام واحد) + 39 قيمة أخرى مستقلة في bayan* (نظام ثانٍ منفصل)
Tokens      → Tier-1 (raw) ✅ يُستخدم / Tier-2 (semantic) ⚠️ جزئي / Tier-3 (component) ❌ معرّف، صفر استخدام
Utilities   → .page-wrap, .text-gray, .green-text... (تُستخدم عبر كل الصفحات تقريبًا)
Atoms       → action-btn, type-badge, .field, .spinner, icon-btn
Molecules   → summary-card family (بعد التوحيد)، stat-mini، info-card
Organisms   → Navbar/Sidebar (nav.js)، Modal system (shared.js)، Table (3 تطبيقات)، AI Assistant
Templates   → Management pages / Statement pages (bayan* — self-contained) / Dashboard
Pages       → 31 ملف (منهم 1 فاضي تمامًا: activity.html)
```

# PHASE 8 — Visual Architecture

**إزاي الأنماط بتتدفق فعليًا (مش نظريًا):**
1. `:root` في `styles.css` بيعرّف 3 طبقات (raw → semantic → component) — **البنية على الورق سليمة**.
2. لكن القواعد الفعلية (`.action-btn`, `.summary-card`, `.field`...) بترجع لـ Tier-1 مباشرة (`var(--border)`, `var(--card-bg)`) **وبتتخطى Tier-2 وTier-3 بالكامل تقريبًا**. الاستثناء الوحيد المؤكد: التوكينز الجديدة (`--card-radius-*`, `--modal-width-*`) المضافة هذه الجلسة، وهي فعلاً مُستخدمة في قواعدها.
3. صفحات `bayan*` بتكسر التدفق بالكامل — عندها `<style>` مستقل و`:root` منفصل، **صفر اتصال** بملف `styles.css` الرئيسي. ده مسار تصميم مواز كامل، مش مجرد استثناء بسيط.
4. الألوان الـ hardcoded (424 استخدام) بتتدفق **برّه** نظام CSS كليًا — قيمة مكتوبة مباشرة داخل HTML/JS، سواء كانت نسخة من متغير موجود (60.4% من الحالات) أو قيمة جديدة تمامًا غالبيتها من نظام `bayan*` المنفصل.
5. الجداول: `.tbl` بترجع لمتغيرات الثيم بشكل سليم، لكن `.data-table` (بلا تعريف) والجدول الـ inline في `dashboard.html` **بيطلعوا برّه التدفق كليًا** — إما بلا شكل، أو بأنماط مكتوبة يدويًا لمرة واحدة.

**فين التجريد (Abstraction) موجود وشغال:** action-btn، type-badge، آلية showModal/closeModal، .page-wrap، .spinner، .empty-state.

**فين التجريد بيتكسر:** لحظة الانتقال من "التعريف" (Tier-3 tokens) إلى "التطبيق" (القواعد الفعلية) — التجريد موجود نظريًا وميتفعّلش عمليًا في أغلب الحالات.

**فين التكرار بيبدأ:** أي مرة صفحة جديدة احتاجت "صندوق بخلفية وحواف مدورة" (كارت) أو "جدول بيانات" وملقتش مرجع واضح تتبعه، فعرّفت كلاس جديد بدل ما ترجع لمكوّن موجود. ده حصل 9 مرات مع الكروت و3 مرات مع الجداول.
